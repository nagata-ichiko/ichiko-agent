# Design — デザインを作り込み、トークン化して build へ渡す

`mock` が「速い認識合わせ（ラフでよい）」なら、`design` は「作り込み＋標準化」。**見た目を仕様として固定**し、build が同じ見た目を再現できるようにする（視覚が run 毎にブレる #6 対策）。Canva MCP は使わず、Claude の Artifact 生成を主軸にする。

## いつ mock でなく design か

| | mock（Sketch） | design（Sketch 深掘り） |
|---|---|---|
| 目的 | 認識合わせ・イメージ一致 | 作り込み・視覚の標準化 |
| 忠実度 | ラフ（合意に必要な粒度） | 高忠実度（実装に落とせる粒度） |
| 成果物 | 画面モック Artifact | 高忠実度 Artifact + **デザイントークン** |
| 次 | plan / design | build（トークンに従って実装） |

→ mock で画面と状態の認識が合ったら design に進む。既存画面の単純な見た目差し替えは mock モードB のままでよい。

## モードA: デザイン作り込み（新規）

1. **前提を読む**: 要件カード（`docs/specs/[SPEC-ID].md`）の受入基準・画面遷移表・§9モック欄、および mock の Artifact を確認する
2. **デザイン方針を1行宣言**: CLAUDE.md「デザイン方針」があれば従う。無ければ用途から宣言（管理画面=使いやすさ優先/LP=訴求優先 等）してから作る（判断のブレを残さない）
3. **高忠実度 Artifact を生成**（自己完結HTML/CSS、CSP制約: 外部CDN/フォント/画像不可・全てインライン or data URI → Artifact ツール）
   - 主要画面 × 主要状態（空/エラー/ローディング/一覧あり）を実装可能な粒度で作る
   - 要件カードの画面遷移表に載る画面を網羅する（表に無い画面は作らない＝許可リスト）
4. **反応を受けて修正 → 再提示**を、合意まで反復する
5. **デザイントークンを抽出**（下記フォーマット）し、`docs/design/[SPEC-ID]-tokens.md` に保存する
6. 要件カード §9 モック欄に Artifact リンクとトークンファイルのパスを記録する
7. 次を提案:「build でこのトークンに従って実装しますか？」

## モードB: デザインシステム / スタイルガイド

プロジェクト横断の視覚標準を作る（1機能でなく全体）。

1. 既存画面・既存CSS（あれば）から現状のトークンを吸い出す。無ければ方針から新規に定義する
2. トークンを `docs/design/design-system.md` に確定する（下記フォーマット）
3. 代表コンポーネント（ボタン/入力/カード/テーブル/トースト）の見本を Artifact で提示する
4. 次を提案:「各画面をこのシステムに揃えますか？（mock モードB / build）」

## デザイントークン フォーマット（固定・build が真実として参照）

```markdown
## カラー
| トークン | 値 | 用途 |
|---------|-----|------|
| color.primary | #2563eb | 主要アクション |
| color.danger | #dc2626 | 破壊的操作・エラー |
| color.text | #111827 | 本文 |
| color.bg | #ffffff | 背景 |

## タイポグラフィ
| トークン | サイズ/太さ | 用途 |
|---------|-----------|------|
| font.h1 | 24px/700 | 画面タイトル |
| font.body | 14px/400 | 本文 |

## スペーシング / 角丸
| トークン | 値 |
|---------|-----|
| space.sm / md / lg | 8 / 16 / 24 px |
| radius.md | 8px |

## コンポーネント規約
- ボタン: primary=塗り / secondary=枠線。高さ40px、角丸 radius.md
- 入力: 枠線 color.text 20%、エラー時 color.danger
<!-- 実装後この表と照合して視覚の正誤を判定する。表に無いスタイルは追加しない -->
```

## ルール

- **ロジックを作らない**（design は見た目とトークンに専念）。ロジック実装は build へ
- Artifact は自己完結（CSP: 外部依存不可、全てインライン or data URI）
- トークンは build にとっての「視覚の仕様」。トークンに無い色・サイズを実装で勝手に足さない（許可リスト方式・視覚ドリフト防止）
- デザイン方針が未確定なら1行で宣言してから進む（未決を残さない）
- Canva 等の外部デザインツール連携が要る場合はユーザーに確認してから（既定は Claude Artifact 生成）
