# リクエストの受け皿と振り分け

内容が確定できないリクエストを受け取り、即答するか5フェーズスキルへ振り分ける。冒頭で「読込メモリ N件」を1行報告する（principles §1）。

## Step 1: 入力の正規化

- **GitHub Issue URL / 番号**: `gh issue view` で body と comments を取得する
- **口頭・テキストのみ**: そのまま分類へ

## Step 2: 分類して振り分け

| カテゴリ | 判断基準 | 対応 |
|---------|---------|------|
| **即答** | 短い質問、コード調査不要 | そのまま簡潔に回答 |
| **調査・回答** | 機能・コード・設計への質問 | コード/設計を Read・Grep し **file:line 引用**で回答（principles §2） |
| **相談・壁打ち** | 方針の議論、意見を求められている | 選択肢を整理し推奨を提示。結論を急がない |
| **既存コード把握** | 「構造を教えて」「overview 作って」 | → `map` |
| **新機能・要件を作る** | まだ無い機能、要件が曖昧 | → `grill`（要件カード） |
| **画面イメージ合わせ / 見た目変更** | モックが欲しい、デザイン差し替え | → `mock` |
| **デザイン作り込み / システム化** | 高忠実度デザイン・トークン・スタイルガイド | → `design` |
| **進め方の計画** | タスク分解・マイルストーン化 | → `plan` |
| **実装・修正・仕様変更・テスト・緊急修正** | コードを書く/変える | → `build` |

- 特定スキルに確定できるなら直接そのスキルを発火する（handle を経由しなくてよい）
- 判断に迷う場合は「これは○○ということでよいですか？」と確認してから振り分ける

## Step 3: 後処理

- **Issue 起点**: 作業完了後、Issue を閉じるかユーザーに確認する
- 完了後は次のフェーズを提案する（principles §1「次のステップ提案」）

## ルール

- **分類を明示する**: 「これは○○と判断し、△△スキルで進めます」と伝えてから動く
- **実装はプランを経由**: コード変更は要件カード/ goal-board（またはスコープ復唱）なしに書き始めない
- **判断に迷ったら聞く**: 無理に分類せずユーザーに確認する
