# Grill — 要件を問い詰めて要件カードにする

ゴールは **「これを読めばゴールが分かる・情報不足ゼロ」の要件カード1枚**（`docs/templates/spec-card.md` 形式）を確定すること。仕様が真実（実装後にこのカードと照らせば正誤を判定でき、開発者への質問が不要になる）。詳細実装（HOW）はコードとテストが真実で、詳細設計書は作らない。

**到達基準（Fix）**: 要件カードの「0. 完成チェックリスト」が全て `[x]`、かつ**(a)解消必須の未決 = なし**。これを満たすまで grill を終えない（(a)が残る限り Build には進めない）。(b)外部確認待ちの未決は「誰の何の確認か」を明記すれば残してよい（該当マイルストーンのみブロック → Step 5）。

## Step 1: 原典スコープ宣言（着手前・P1手戻り防止）

入力（会話・Issue・既存資料・デザイン）が**扱っている関心事**を先に宣言する: `{画面 / 帳票 / DB / API / バッチ}` のどれか。要件カードのメモ欄に記録する。

- 原典が触れていない関心事（特に DB/テーブル/カラム/DDL）を**推測で創作しない**。既存流用が既定
- 宣言外の領域に踏み込む必要が出たら「未決」に上げてユーザーに確認する

## Step 2: ドメイン用語の固定（P2取り違え防止）

- `docs/domain-knowledge/` があれば先に Read し、用語・**対義語ペア**（受入/差入・貸手/借手 等）・実装状態を確認する
- 取り違えやすい語は、採用した解釈を1行で宣言してから進む（要件カードのメモ欄に記録）
- 新たに確定した用語・対義語は `docs/domain-knowledge/` に file:line 付きで追記する（2回調べた知識は必ず固定）

## Step 3: 既存コードの現状把握（変更系のみ・矛盾の早期発見）

既存機能の変更要件では、**問い詰めの前に該当既存コードを読む**（会話と実コードの矛盾を build まで持ち越さない）:

- 該当箇所を特定して Read し（`.claude/skills/_shared/code-search-2stage.md` の2段階探索）、現在の挙動・制約を **file:line 付きで要件カードのメモ欄に記録**する
- 会話で合意しつつある内容と既存コードの実態が矛盾したら、**要件確定前に指摘**し、採用する解決（既存に合わせる / 既存を変える）をカードに記録する
- 新規機能（変更対象の既存コードなし）はこの Step をスキップし、カードのメモ欄に「新規（変更対象なし）」と明記する

## Step 4: 機能タイプを判定し、必須チェックリストを全て潰す

まず機能タイプを判定する（複数該当あり）。**該当タイプの必須項目を全て埋めるまで問い詰める**（「該当するもののみ選ぶ」ではなく、タイプが決まれば必須項目も決まる＝選定裁量を無くす）。

### タイプ判定 → 必須項目

| 機能タイプ | 判定 | 必ず埋める要件カードのセクション | 追加で必ず聞くこと |
|-----------|------|-----------------------------|------------------|
| **UI/画面あり** | 画面・フォーム・一覧がある | 4.画面遷移表 | 全遷移（成功/失敗/キャンセル）、空/エラー/ローディング状態、遷移ガード（権限・前提） |
| **状態を持つ** | エンティティが status を持つ・ワークフローがある | 5.状態遷移表 | 全状態と全イベント、無効遷移の扱い、各遷移の副作用（通知・記録）、許可アクター |
| **入力を受ける** | フォーム・API リクエスト・取込がある | 6.入力・バリデーション表 | 各項目の型/必須/制約（文字数・範囲・一意性）、違反時の挙動、重複・上限超過時 |
| **権限がある** | ロール別に可否が変わる | 3.受入基準 に権限条件 | ロール別の可否、データ可視範囲、未権限時の挙動（403/非表示） |
| **データ削除がある** | 削除操作がある | 3.受入基準 / 6.入力表 | 論理/物理、連鎖削除の範囲、削除後の参照 |
| **同時操作がありうる** | 複数人が同一データを操作 | 3.受入基準 / 5.状態遷移表 | 先勝ち/後勝ち/排他、競合時の挙動 |
| **API/バッチ（UIなし）** | エンドポイント・定期実行 | 3.受入基準 に I/O 契約 | 入出力スキーマ、異常系（タイムアウト・部分失敗・冪等性）、実行契機 |
| **外部サービス連携がある** | 外部API呼び出し・Webhook・外部SaaSの設定変更を伴う | 3.受入基準 / 10.メモ | 連携先サービスと認証方式・必要スコープ、連携失敗時の挙動（リトライ・通知・フォールバック） |
| **環境変数・secrets が増える** | 新しい環境変数・APIキー・接続情報を導入する | 6.入力表 / 10.メモ | 変数名、供給元（.env / CI secrets 等）、未設定時の挙動（起動失敗 or 機能無効） |
| **エラー・リトライ・キャッシュの方針が要る** | 外部I/O・失敗しうる処理・重い処理がある | 3.受入基準 | エラー時にユーザーへ何を見せるか、リトライ（回数・間隔）、キャッシュの有無と失効条件（implementation.md の例外時既定に暗黙で流さない） |
| **性能・件数の目標がある** | 応答時間・処理件数・同時利用数の言及がある | 3.受入基準 | 数値目標（例: 応答200ms以内・1万件/日）と検証手段（→ testing.md） |
| **UIあり かつ デザイン方針が未定義** | CLAUDE.md「デザイン方針」が未記入 | 10.メモ ＋ CLAUDE.md | デザイン方針（管理画面=使いやすさ優先 / LP=訴求優先 等）を確定し CLAUDE.md「デザイン方針」に記入 |

### 問い詰め方

- **Yes/No か選択肢**で答えやすく聞く。1回の応答で複数論点を束ねてよい（往復を減らす）
- ユーザーにフォーマット記入を求めない。会話から引き出して**AIが表に落とす**
- **推測で埋めない**。ただし潰し切るのが原則。どうしても決まらない論点だけ「未決」に残す（後述 Fix ゲートで、未決が残る限り Build に進めない）
- 「適切に」「いい感じに」等の曖昧語は具体値（数値・条件）に変換してから記入する

## Step 5: 要件カードを書く（固定フォーマット）

`docs/specs/[SPEC-ID].md` に `docs/templates/spec-card.md` 形式で1枚生成する（SPEC-ID は `SPEC-<領域>-NNN`）。

- 全セクション必須。該当しないセクションも消さず「なし（理由）」と明記して表を削除する
- **受入基準は Given-When-Then**（テストの元ネタ → testing.md）
- **画面遷移表・状態遷移表・入力表は Step 4 のタイプ判定で必須になったものを埋める**（表に無い遷移は禁止＝許可リスト）
- スコープ外を明示（最小変更原則）
- モック欄は Sketch（mock スキル）で埋める

### 「未決」の2分類（§8 に書き分ける）

潰し切るのが原則だが、どうしても残る論点は種別を明記する。種別で Build 可否の扱いが変わる（Step 6・build 側と接続）:

- **(a) 解消必須**: grill 内で自分／ユーザーに聞けば潰せる論点。**Build 前に必ずゼロにする**（残したまま次へ進まない）。
- **(b) 外部確認待ち**: ユーザーでも即答できずビジネス／外部関係者の確認が要る論点。該当する**マイルストーンだけをブロック**し、依存しない部分の Build は進めてよい。§8 に「(b) 外部確認待ち: 〜（誰の確認が必要か）」と明記する。

## Step 6: Fix ゲート判定と次ステップ

1. **完成チェックリスト（カード §0）を検証**する。全項目 `[x]`・**(a)解消必須の未決=なし** を満たすか自己確認する
   - (a)解消必須が残る → 不足項目を再度問い詰める（Step 4 に戻る）。**(a)を残したまま完了報告しない**
   - (b)外部確認待ちのみ残る → 「誰の何の確認が必要か」を明示してユーザーに提示。依存しないマイルストーンは Build へ進めてよい旨を添える
2. 要件カードをユーザーに提示し、受入基準・遷移表・スコープ外に齟齬がないか確認する
3. コミット: `docs: [SPEC-ID] 要件カード作成`
4. 次を提案:
   - UIがある → 「mock で画面モックを作って認識合わせしますか？」（Sketch）
   - 認識が揃っている → 「plan でマイルストーン分解に進みますか？」（Plan）

## ルール

- 受入基準・スコープ・遷移表の言語化は AI が主導する。ユーザーには判断だけを求める
- 要件カードは WHAT（何を・なぜ・どの遷移が正か）に集中。HOW（実装詳細）は build がコードで確定する
- 1機能 = 1カード。巨大な要件は複数カードに割る
- **(a)解消必須の未決を残したままフェーズを終えない**。残るなら「まだ確定していない」と明示し、ユーザーの判断を仰いで潰す。(b)外部確認待ちは「誰の何の確認か」を明記して残してよい
