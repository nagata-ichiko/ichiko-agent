# テスト戦略

**テスト＝仕様**。要件カードの受入基準（Given-When-Then）をテストの元ネタにし、実装コードの分岐・エッジケースで補完する。テスト件数は機械取得（実行結果）で数える。

| テスト種別 | ツール例 | 必須/推奨 |
|-----------|---------|----------|
| ユニット | vitest / jest / pytest | 必須 |
| E2E | Playwright / Cypress | 必須（UIあり） |
| API コントラクト | openapi-validator / dredd | 推奨 |
| 負荷 | k6 / artillery | 非機能にSLO記載があれば条件付き必須 |

- 非機能に数値目標（例「1000req/s」「応答200ms以内」）があれば、その検証手段も実装に含める
- E2Eフレームワーク検出: `cypress.config.*` → Cypress / `playwright.config.*` → Playwright / どちらも無ければ Playwright を既定として提案

## テストケース設計

- 期待値が不一致でも**すぐテストを修正しない**。設計・実装・テストの三者を突き合わせ、障害かテスト誤りかを判断する

## 素通り（Silent PASS）の禁止

データ不在・条件未達で早期 return して素通り PASS するテストを書かない。

```ts
// NG: データがなければ何も検証せず PASS
if (rows.length === 0) return;
// OK: データ不在はテスト環境の問題として明示的に FAIL
expect(rows.length, '前提データが存在しません').toBeGreaterThan(0);
```

`test.skip` / `if (!data) return` は同種の問題。前提データが必要ならセットアップで作成するか `expect` で FAIL させる。ユーザーが明示承認した場合のみ例外。

## タイムアウト変更ポリシー

タイムアウト延長は**最後の手段**。ユーザー承認なしに変更しない。

1. まず根本原因を調査（遅い処理・不正データ・ロジックバグのサイン）
2. 原因を修正
3. 解決しない場合のみユーザーに確認 → 変更時は理由をコメントに残す

目安: 個別ケース30秒超・beforeAll/afterAll 60秒超は要調査。数分以上はほぼ確実にロジック/データの問題。

## E2Eデグレチェック（実装完了時）

フロント/バックのソース変更を伴う作業が完了したら、**PR作成（または最終コミット）前にE2E全件を実行**しデグレなしを確認する。マイルストーン単位の中間コミットはE2E全件を待たず積んでよいが、全パスするまでPRを作成しない。失敗が既存の flaky なら切り分けてユーザーに報告。ドキュメントのみの変更では不要。
