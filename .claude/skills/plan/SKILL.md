# Plan — ゴールをマイルストーンに割る

要件カード（grill 成果物）を、依存順に並んだマイルストーンへ分解し `goal-board.md` に落とす。旧 orchestrate の3層PM儀式は使わない — 分解はここ1箇所で行い、実装は build が自律で回す。

## Step 1: 入力を読む

- 要件カード `docs/specs/[SPEC-ID].md`（受入基準・画面遷移表・状態遷移表・スコープ外）。**Fix ゲート未達（(a)解消必須の未決あり・完成チェックリスト未達）なら分解に入らず grill に差し戻す**。(b)外部確認待ちの未決のみなら分解に入ってよい（Step 2 でブロック印を付ける）
- 関連する既存コードの構造（不明なら `map` スキルで軽量把握してから戻る）
- モックがあれば画面・状態の一覧

## Step 2: マイルストーンに分解

- ゴールを**依存順**のマイルストーンに割る（1マイルストーン = 1コミット単位が目安）
- 各マイルストーンに**完了判定**（テストが通る/画面が動く 等の具体条件）を書く
- 独立して並列実行できるマイルストーンは印を付ける（build が parallel-work で fan-out する判断材料）
- 要件カード §8 の (b)外部確認待ちに依存するマイルストーンには `[B: 誰の何の確認待ちか]` を付ける（build はそのマイルストーンだけブロックし、依存しないものを進める）
- 流用すべき既存コードを file:line で挙げる
- やらないこと（スコープ外の理由 = WHY NOT）を残す

## Step 3: goal-board.md を書く

`impl-plans/[SPEC-ID].md`（または作業識別子）に `docs/templates/goal-board.md` 形式で生成する:

```markdown
# ゴール: <一文で>
## 受入基準（ゴール到達の判定）
## マイルストーン
- [ ] M1: <名前> — 完了判定: <条件> — 状態: 未着手
  - [ ] タスク1.1
## 現在地 / ブロッカー / 次アクション
```

goal-board は**生きたドキュメント**。build が進捗・ブロッカー・現在地を随時更新する。作業完了後は削除してよい（永続文書ではない）。

## Step 4: 確認と次ステップ

- 分解結果（マイルストーンと完了判定・並列可否）をユーザーに提示する。これがスコープ復唱を兼ねる（principles §7）
- 承認を得たら次を提案:「build で実装ループに入りますか？」（Build）

## ルール

- 分解は依存関係を最優先。依存元マイルストーンを先に完了させる
- 完了判定が書けないマイルストーンは粒度が粗すぎる — さらに割る
- DBスキーマ変更を含むマイルストーンは並列に載せない（parallel-work の禁止事項）
