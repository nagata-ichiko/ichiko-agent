# Build — 封筒の中で自律実装する

**着手前に `.claude/rules/autonomous-mode.md` を必ず読む**（実行規律: スコープ分割・逐次commit・ポーリング禁止・委任信頼性）。実装しながら詳細を確定する — 詳細設計書は作らず、コードとテストを真実にする。

## 準備フェーズ（コードに触れない）

1. **要件カード**（`docs/specs/[SPEC-ID].md`）を読み、**Fix ゲートを検証**する。無ければ「先に grill で要件カードが必要です」と提案。
   - **Fix ゲート**: 「0. 完成チェックリスト」が全て `[x]`、かつ「未決 = なし」であること。**未決が残る／チェックリスト未達なら Build に入らず grill に差し戻す**（実装中に開発者へ質問する状況を構造的に無くす）
   - 受入基準・画面遷移表・状態遷移表・スコープ外を把握する。**表に無い遷移は実装しない**（許可リスト方式）。実装後はこの表と照合して正誤判定する
2. **goal-board**（`impl-plans/[SPEC-ID].md`）を読む。無く、かつ3ファイル超の変更なら「先に plan で分解しますか？」と提案（小修正はスキップ可）
3. **既存コードを把握**（`.claude/skills/_shared/code-search-2stage.md` の2段階探索）。流用すべき既存コード・共通コンポーネントを特定し、再発明しない
4. ブランチを作成（`git-workflow.md` の命名: `feat/… | fix/…`）。緊急修正は下記モードC

## 実装ループ（マイルストーンごと）

各マイルストーンで以下を回し、**完了ごとにコミット**する（最後にまとめない）:

5. **テスト先行（推奨）** — 受入基準（Given-When-Then）から先にテストを書く（`testing.md`）。素通りPASS禁止
6. **実装** — goal-board の順序で書く。判断基準:
   - **本体が直接実装**（既定）: 途中判断が要る、既存コードと矛盾する、エッジケースが曖昧な場合
   - **coder に委任**: マイルストーンの手順が明確で途中判断が不要な場合。`Agent(subagent_type: "coder")` に goal-board のパスと対象ファイルを渡す
   - **並列 fan-out**: 独立したマイルストーンが複数あれば `parallel-work.md` に従い worktree 分離で coder を並列起動（DBスキーマ変更は並列禁止）
7. **テスト実行** — 該当範囲のテストを通す。通らなければ3回まで修正、ダメなら失敗内容を添えて報告（`implementation.md`）。出力はファイルへ、context には要約だけ
8. **自己レビュー（本体が直接・委任不可）** — `.claude/skills/_shared/review-checklist.md` の観点で確認、指摘基準は `review-standards.md`（信頼度80以上のみ修正対象）。指摘→修正→再レビュー（最大3ループ）
9. **コミット** — マイルストーン単位。goal-board の状態を更新（現在地/ブロッカー/次アクション）

## モードC: 緊急修正（hotfix）

本番障害・脆弱性の即時修正でユーザーが「緊急」と明示した場合のみ、コード先行を許可する（設計書ファーストの例外）:

- ブランチ `fix/…`（または `hotfix/…`）で最小修正 → テスト → コミット
- **修正後に要件カード/ goal-board を追従更新**する（後追いでの整合が義務）

## 仕上げ

10. 全マイルストーン完了後、`.claude/skills/_shared/finish-impl.md` の共通仕上げ手順を実行
11. E2Eデグレチェック（`testing.md`: フロント/バック変更時はコミット前にE2E全件）
12. 要件カードの実装状態を更新。goal-board は完了したら削除してよい
13. 次を提案（「PRを作りますか？」「次のマイルストーン/機能に進みますか？」「テストを補強しますか？」）

## ガードレール（越権防止・実ログP0）

- 封筒の一線（外部ネット・データ送出・不可逆破壊）を越える操作は実行前に確認（`principles.md` §11 / `security.md`）
- deny・フック・ブランチ保護に当たっても回避策を積み重ねない。1回確認 or ブロッカー明示スキップ（`autonomous-mode.md` §5）
- 履歴改変（force-push/reset）は自動バックアップ後に実行（`git-workflow.md`）

## ルール

- 要件カードの受入基準に無い機能を実装しない（スコープ厳守）。画面遷移表・状態遷移表に無い遷移は実装しない（許可リスト方式）
- Build 着手の前提は Fix ゲート通過（未決=なし・完成チェックリスト全達）。未達なら grill に差し戻す
- レビュー→修正ループは最大3回。PR作成はユーザー承認後
