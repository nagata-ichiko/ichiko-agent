# Grill — 要件を問い詰めて要件カードにする

ゴールは「作る前に認識を一致させる」こと。AIが主導して問い詰め、曖昧さを潰し、**要件カード1枚**（`docs/templates/spec-card.md` 形式）を確定する。詳細設計書は作らない（詳細はコードとテストが真実）。

## Step 1: 原典スコープ宣言（着手前・P1手戻り防止）

入力（会話・Issue・既存資料・デザイン）が**扱っている関心事**を先に宣言する: `{画面 / 帳票 / DB / API / バッチ}` のどれか。

- 原典が触れていない関心事（特に DB/テーブル/カラム/DDL）を**推測で創作しない**。既存流用が既定
- 宣言外の領域に踏み込む必要が出たら、要件カードの「未決」に上げてユーザーに確認する

## Step 2: ドメイン用語の固定（P2取り違え防止）

- `docs/domain-knowledge/` があれば先に Read し、用語・**対義語ペア**（受入/差入・貸手/借手 等）・実装状態を確認する
- 取り違えやすい語は、採用した解釈を1行で宣言してから進む
- 新たに確定した用語・対義語は `docs/domain-knowledge/` に file:line 付きで追記する（2回調べた知識は必ず固定）

## Step 3: 問い詰め（最大8問・Yes/Noで答えやすく）

不足情報を質問する。該当するカテゴリのみ:

- **バリデーション**（文字数上限・入力フォーマット・許容範囲）
- **エラー・異常系**（重複・上限超過・期限切れ時の振る舞い）
- **権限・アクセス制御**（ロール別可否・データ可視範囲）
- **状態遷移**（遷移制約・無効遷移の扱い）
- **同時操作**（先勝ち/後勝ち/排他）
- **データ削除**（論理/物理・連鎖削除）
- 担当者（未指定なら「未定」）

ユーザーにフォーマット記入を求めない。会話から直接引き出す。推測で埋めず、潰しきれない論点は「未決」に残す。

## Step 4: 要件カードを書く

`docs/specs/[SPEC-ID].md` に `docs/templates/spec-card.md` 形式で1枚生成する（SPEC-ID は `SPEC-<領域>-NNN`）。

- **受入基準は Given-When-Then** で書く（これがテストの元ネタになる → testing.md）
- スコープ外を明示する（最小変更原則）
- 未決・対義語ペアの注記を残す
- モック欄は Sketch（mock スキル）で埋める

## Step 5: 確認と次ステップ

- 要件カードをユーザーに提示し、受入基準・スコープ外・未決の齟齬がないか確認する
- コミット: `docs: [SPEC-ID] 要件カード作成`
- 次を提案:
  - UIがある → 「mock で画面モックを作って認識合わせしますか？」（Sketch）
  - 認識が揃っている → 「plan でマイルストーン分解に進みますか？」（Plan）

## ルール

- 受入基準・スコープの言語化は AI が主導する。ユーザーには判断だけを求める
- 要件カードは WHAT（何を・なぜ）に集中。HOW（実装詳細）は build がコードで確定する
- 1機能 = 1カード。巨大な要件は複数カードに割る
