# M3構成レビュー — 「エンジニアがそのまま働く」理想像との差分

**日付**: 2026-07-01 / **対象**: feat/harness-m3 の新ハーネス / **観点**: ユーザー提示の理想像
**理想像**: ゴール確定 → フォーマット準拠の仕様書（画面遷移・状態遷移が明確な形式で drift しない）→ 実装後は仕様書だけで判断可 → 開発者に質問不要 → エンジニアがそのまま働く感じ

---

## 結論（3行）

- **並列動作**: ✅ 十分。build/map/review-pr が fan-out、parallel-work が規律。
- **毎回同じ進め方**: △ フェーズ骨格（Grill→Sketch→Plan→Build）は一定。ただし各スキルは散文ガイドで、ステップ内の振る舞いはモデル解釈に委ねられ run 毎にブレうる。
- **仕様書だけで判断・質問不要**: ✗ 未達。(1) 画面遷移/状態遷移の drift-proof フォーマットが無い、(2) 要件カードに「未決」を残せる＝質問が実装まで持ち越せる、(3) M3は「コードが真実」で理想（仕様が真実）と逆。

## 観点別評価

### Q1. 方針をAIが独自解釈する必要がある箇所はないか → **複数ある**

| 箇所 | 独自解釈の内容 | リスク |
|---|---|---|
| 要件カード `未決` 欄 | 潰しきれない論点を残置＝実装時にAIが解釈 | 質問不要・仕様のみ判断に反する |
| grill Step3 質問カテゴリ | 「該当するもののみ」をAIが選ぶ・最大8問をAIが選定 | 抜け漏れが run 毎に変わる |
| grill Step1 原典スコープ宣言 | 入力の関心事をAIが判定 | 判定ブレ |
| build Step6 委任判断 | 本体直実装/coder委任/並列fan-out を「自動判断」 | 同じ作業でも経路が変わる |
| build Step2 | 「3ファイル超なら plan 提案、小修正はスキップ可」 | 境界がAI裁量 |
| mock モードA | 「デザイン指定が無ければ CLAUDE.md 方針」「管理画面は使いやすさ優先」 | 見た目がブレる |
| implementation.md 既定判断表 | 「仮採用」= 不足時にAIが推測 | 仮採用の質が run 依存 |
| handle 分類 | リクエストをカテゴリに分類してルーティング | 分類ブレ |
| 画面遷移/状態遷移 | **記録フォーマットが無い**ため表現方法をAIが毎回決める | **最大のドリフト源** |

→ フェーズの入り口（どのスキルへ）はほぼ決定的だが、**各スキルの内側は解釈余地が残る**。特に遷移系はフォーマット不在で無限にブレうる。

### Q2. 毎回同じ仕事の進め方をするか → **フェーズ単位=Yes / ステップ単位=No**

- 5フェーズの直列（Grill→Sketch→Plan→Build、既存把握は Map）は principles.md と各スキルで一定。
- ただしスキル本文は「推奨/目安/迷ったら/スキップ可」等の柔らかい語を含み、**強制ゲート（この条件を満たすまで次に進めない）が少ない**。
- 決定性を上げるには: (a) 必須フォーマット（穴埋め式）、(b) 明示ゲート（例: 未決=0 でなければ Build 不可）、(c) 完了必須チェックリスト、が要る。現状は「半構造」。

### Q3. 並列に動作できるか → **✅ 十分**

- build: 独立マイルストーンを parallel-work に従い worktree 分離で coder に fan-out（DBスキーマ変更は並列禁止の安全弁あり）。
- map: 200ファイル超で内蔵 Explore を並列起動。
- review-pr: 5観点を並列。
- parallel-work.md: 委任信頼性（証跡付き報告・親の独立検証・ポーリング禁止）を規定済み。
- 補足: 決定的オーケストレーション（Workflow的な台本）ではなくモデルがエージェントを起動する方式。台本化すればさらに再現性は上がる。

## 最大のギャップ：真実の所在と遷移フォーマット

### ギャップA. 「コードが真実」 vs 「仕様書が真実」

M3（Loop Engineering）は **詳細はコードとテストが真実** を採り、要件カードは軽量1枚にした。これは実ログ分析（713本）で判明した BDUF のコンテキスト破綻・spec過剰生成（原典に無いテーブルを創作）への対策だった＝根拠がある。

一方あなたの理想は **仕様書を見るだけで判断できる**＝仕様が真実。実装後の判断・質問不要を担保するには、仕様が**完全かつ曖昧さゼロ**である必要がある。ここは設計思想の分岐点。

### ギャップB. 画面遷移・状態遷移の drift-proof フォーマットが無い（最重要）

M3 で `screen-transition-diagram.md` と phase テンプレを削除した結果、**遷移を記録する決まった場所と形式が消えた**。要件カードは自由記述で、遷移はモデルが毎回好きに表現する＝あなたが最も嫌う「ブレ」が構造的に起きる。

---

## 推奨：リーンを保ったまま「ブレる所だけ」を固める（BDUF復活はしない）

29テンプレのBDUFには戻さない。代わりに、**ドリフトが実害を生む2点（画面遷移・状態遷移）だけを必須フォーマット化**し、Fix ゲートを厳格化する。

### 提案1. 要件カード v2 — 遷移を必須の表形式に

`spec-card.md` に、UI/状態を持つ機能では必須の構造化セクションを追加（表なのでブレない・埋まっていなければ未完）:

```markdown
## 画面遷移（UIがある場合は必須）
| 遷移元画面 | トリガー(操作/イベント) | 遷移先画面 | 条件・ガード |
|-----------|----------------------|-----------|-------------|
| 一覧 | 「新規」クリック | 作成フォーム | ログイン済み |
| 作成フォーム | 保存成功 | 一覧(トースト) | バリデーションOK |

## 状態遷移（状態を持つエンティティは必須）
| 現在状態 | イベント | 次状態 | 副作用 | 許可アクター |
|---------|---------|-------|-------|------------|
| 下書き | 申請 | 承認待ち | 通知送信 | 作成者 |
| 承認待ち | 却下 | 下書き | 理由記録 | 承認者 |
<!-- 表に無い遷移は「禁止」（implementation.md の許可リスト方式と接続）。実装後この表と照合して判断する -->
```

これで「実装後、この表と照らすだけで正誤判定できる」状態になり、遷移の実装解釈のブレが消える。

### 提案2. Fix ゲート — 未決=0 を Build の前提に

- Build 着手の前提を「要件カードの `未決` が空、受入基準が全て Given-When-Then で埋まっている」に固定。
- 未決が残るなら Build に入らず grill に差し戻す（＝実装中に開発者へ質問する状況を構造的に無くす）。
- principles.md か build/SKILL.md にゲートとして明記。

### 提案3. build の委任判断を決定化

- 「本体直/委任/並列」の自動判断を、goal-board のマイルストーン属性（`[P]`並列可・完了判定の明確さ）から**機械的に決まる**ルールにする（AI裁量を減らす）。

### 提案4. （任意）決定性を上げる仕上げ

- 各スキルの「推奨/目安」を、可能な範囲で「MUST/条件」に置換。
- grill の質問カテゴリを「機能タイプ別の必須チェックリスト」にして選定裁量を減らす。

## 優先度の提案

1. **提案1・2（要件カードv2 + Fix ゲート）** — 理想像への効果が最大。M4 の前に入れる価値あり（"M3.5"）。
2. **M4（自律の封筒の強制）** — セキュリティP0。独立して重要。
3. 提案3・4 — 決定性の底上げ。M5 ドッグフーディングで実際のブレを観測してから調整が効率的。

> 判断ポイント: 「仕様が真実（提案1・2で仕様を完全化）」に寄せるか、「コードが真実」を保ちつつ遷移だけ固めるか。後者（ハイブリッド）を推奨 — リーンさとブレ防止を両立できる。

## 確定した次アクション（2026-07-01 ユーザー承認）→ M3.5

ユーザーの方針決定。次セッションで着手する。

1. **grill（grill me）はこのまま良い** — 問い詰めの仕組みは維持。
2. **仕様書を明確な固定フォーマットにする（本丸）** — 自由フォーマットはブレるので廃す。到達目標 = **「情報不足がなく、それを読めばゴールが分かる状態が完成」**。
   - 要件カードを固定フォーマット化（自由記述の余地を減らす）。
   - 画面遷移・状態遷移を明確な表形式で必須化（提案1）。
   - 「仕様が真実」寄りに倒す（読めば判断でき、実装中に開発者へ質問しない状態）。
3. **grill で必要な情報を全て集め切る** — 「未決」を残さない方向。Fix ゲート（提案2: 未決=0）を厳格化し、grill の質問網羅性を上げる（機能タイプ別の必須チェックリスト化）。
4. **画面デザイン系スキル + Claude Design（デザインツール）連携を盛り込む** — mock スキルの拡張または新スキル。Canva MCP（接続済み `mcp__claude_ai_Canva__*`）等の外部デザインツール連携で、モックだけでなく実デザイン成果物を扱えるようにする。
5. 実施順: 上記（M3.5 = 仕様フォーマット強化 + grill 網羅性 + デザイン連携）→ その後 **M4（自律の封筒の強制・セキュリティ）**。

**M3.5 で触る想定**: `docs/templates/spec-card.md`（v2化）/ `.claude/skills/grill/SKILL.md`（網羅チェックリスト・Fixゲート）/ `.claude/rules/principles.md` or `build/SKILL.md`（Fixゲート明記）/ `.claude/skills/mock/SKILL.md`（デザインツール連携）or 新デザインスキル。

## M3.5 実施記録（2026-07-01）

仕様フォーマット強化 + grill 網羅性 + Fix ゲートを実装。デザイン連携は**スコープ外に切り出し**（ユーザー判断: Canva MCP は避け、Claude Design 連携か専用スキルを後日検討）。

### 変更内容

1. **`docs/templates/spec-card.md` v2化** — 固定フォーマット化。
   - §0 完成チェックリスト（Fix ゲート）を新設: 全 `[x]`・未決=なし でなければ Build 不可。
   - §4 画面遷移表・§5 状態遷移表・§6 入力/バリデーション表を必須の表形式で追加（表に無い遷移=禁止＝許可リスト方式）。
   - 全セクション必須。該当なしは欄削除でなく「なし（理由）」明記。到達目標「情報不足ゼロ・読めばゴールが分かる=完成」。
2. **`.claude/skills/grill/SKILL.md`** — Step3 を「該当カテゴリのみ選ぶ」から**機能タイプ判定→必須項目が機械的に決まる**方式へ（選定裁量を除去）。7タイプ×必須セクション×追加質問の表。Step5 に Fix ゲート自己検証を追加（未決を残して完了報告しない）。
3. **`.claude/skills/build/SKILL.md`** — 準備フェーズ1に Fix ゲート検証（未達なら grill 差し戻し）、遷移表照合を追加。ルールに Fix ゲート前提とスコープ厳守（遷移許可リスト）を追記。
4. **`.claude/rules/principles.md`** — §1 に「要件カードファースト（仕様が真実）」強化と「Fix ゲート」原則を追加。
5. **`.claude/skills/plan/SKILL.md`** — 入力欄に遷移表・Fix ゲート差し戻しを反映。

### 効果

- 遷移フォーマット不在（最大のドリフト源, ギャップB）を表形式で解消。
- 「未決を実装まで持ち越せる」問題（Q1）を Fix ゲートで構造的に封鎖。
- 「仕様が真実」寄せ完了。実装後はカードと照合して正誤判定でき、開発者質問が不要な設計。
