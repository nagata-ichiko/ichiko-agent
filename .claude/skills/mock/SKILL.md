# Mock — 画面モックで認識を合わせる

「言葉で合意」より「絵で合意」。実際に見えるモックを作ってユーザーと認識をすり合わせる。ロジックは作らない。

## モードA: 認識合わせモック（Sketch フェーズ）

要件カード（grill 成果物）の受入基準を、動く見た目に翻訳して見せる。

1. 要件カード（`docs/specs/[SPEC-ID].md`）があれば読み、画面・主要要素・状態を把握する
2. **Artifact でHTMLモックを生成**する（自己完結HTML/CSS、外部依存なし → Artifact ツール）
   - 主要画面・主要な状態（空/エラー/一覧あり 等）をカバーする
   - デザイン指定（Figma・スクショ）があれば忠実に、なければ CLAUDE.md「デザイン方針」に従う（管理画面は使いやすさ優先、特記なければレスポンシブ）
3. ユーザーの反応を受けて**モックを修正 → 再提示**を、認識が一致するまで反復する
4. 一致したら、要件カードの「モック」欄に Artifact リンクを記録し、受入基準に反映漏れがないか確認する
5. 次を提案:「plan でマイルストーン分解に進みますか？」（Plan）

## モードB: 見た目の差し替え（既存画面）

実装済み画面の見た目だけを変える（ロジックは一切触らない）。

1. 対象画面のコードを特定し、UI部分（HTML/CSS/テンプレート）を把握する
2. デザインファイルがあれば分析、なければ CLAUDE.md「デザイン方針」に従う
3. **見た目部分のみ変更**（バリデーション・API呼び出し・データ処理は変更しない）
4. 既存テストが通ることを確認（通らなくなったら変更を見直す）
5. `.claude/skills/_shared/finish-impl.md` の仕上げ手順 → コミット `feat: [SPEC-ID] デザイン差し替え`
6. 次を提案:「他の画面も差し替えますか？」

## ルール

- モードBではロジックを一切変更しない。ロジック変更が要るなら build へ回す
- モックは認識合わせの道具。作り込みすぎず、合意に必要な粒度で速く回す
- Artifact は自己完結（CSP制約: 外部CDN/フォント/画像不可、全てインライン or data URI）
