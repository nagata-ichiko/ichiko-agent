# Git ワークフロー

## ブランチ戦略

- ベースブランチ: `main`（main 直上での作業開始は避け、必ずブランチを切る）
- 命名: `{type}/{issue番号 or 識別子}-{概要}`
  - 例: `feat/100-delivery-route` / `fix/201-login-error` / `docs/26-overview-format` / `refactor/optimize-user-query`
- 並列作業では全リポで同名ブランチを使う（→ `.claude/rules/parallel-work.md`）

## コミットメッセージ規約

形式: `[type]: [概要]`（Issue/識別子があれば概要に含める）

| type | 用途 |
|------|------|
| `feat:` | 新機能実装 |
| `fix:` | バグ修正・レビュー指摘対応 |
| `refactor:` | 外部仕様を変えない内部改善・コード品質改善 |
| `test:` | テスト追加・修正（`test(ut):` / `test(e2e):` で細分化可） |
| `docs:` | 要件カード・ドキュメント |
| `chore:` | 設定・ビルド・雑務 |

- 自律ループではマイルストーン完了ごとにコミットする（→ `.claude/rules/autonomous-mode.md`）

## 履歴改変（force-push・reset）

- `git push --force` / `git reset --hard` 等の履歴改変は、**実行前に自動バックアップ（バックアップブランチ or タグ）を取ってから実行**する
- バックアップ手順の強制は M4 で hooks 側に実装予定。それまでは手動でバックアップブランチを切ってから実行する
- フック・deny・ブランチ保護を無効化して履歴改変を通すことは禁止（→ security.md / principles.md §11）

## リファクタリング

- ブランチ: `refactor/[対象の簡潔な説明]`
- 外部仕様が変わらないなら要件カード更新は不要。非機能（パフォーマンス等）に影響する場合のみ関連ドキュメントを更新
- **既存テストが全てパスすること**を振る舞い不変の証明とする

## リリース

リリース時（手動またはユーザー指示で）：

1. `git tag vX.Y.Z`（メジャー=破壊的変更 / マイナー=新機能 / パッチ=バグ修正）
2. リリースノートをタグ範囲のコミットから生成（含まれる変更・破壊的変更の有無・マイグレーション手順）

## マルチ開発者での競合回避

- 1人1ブランチを厳守。同じ対象を複数人が同時に触らない
- 追記のみで済むファイル（一覧系）は既存行の変更を避け、Git の自動マージに委ねる
