# エラーリカバリーパターン集

よくある失敗パターンと対処法。問題発生時に参照する。

## 1. テスト失敗

### ユニットテストが通らない
1. エラーメッセージを読み、失敗しているテストケースを特定する
2. テストコードとプロダクションコードの両方を読む
3. 原因を切り分ける:
   - **テストが古い:** Spec変更に追従していない → テストを修正
   - **実装バグ:** ロジックが仕様と異なる → 実装を修正
   - **環境依存:** モック不足、環境変数未設定 → テスト環境を修正
4. 3回修正して通らない場合 → 失敗内容をまとめて人間に判断を仰ぐ

### E2Eテストがタイムアウト
1. dev サーバーが起動しているか確認（CLAUDE.md の開発コマンドを参照）
2. `baseURL` が正しいか確認（playwright.config.ts）
3. 認証バイパスが正しく設定されているか確認
4. ネットワーク依存のテストは `page.waitForResponse()` でAPI応答を待つ

## 2. CI失敗

### GitHub Actions が赤い
1. **どのジョブが失敗したか** をまず確認（unit / e2e / lint / build）
2. ジョブ別の対処:
   - **lint:** ローカルで lint を実行して修正
   - **build:** TypeScript型エラーが多い。`tsc --noEmit` でローカル確認
   - **unit:** ローカルで再現するか確認。CI固有の場合は環境差異を調査
   - **e2e:** CI上のブラウザバージョン差異、またはタイムアウト値の調整
3. `.github/workflows/` の設定を確認（Node.jsバージョン、サービスコンテナ等）

### 依存パッケージのインストール失敗
1. `package-lock.json` / `yarn.lock` が最新か確認
2. Node.jsバージョンが `.nvmrc` と一致しているか確認
3. プライベートレジストリの認証が設定されているか確認

## 3. 要件カード⇔コード不整合

### 要件カードの受入基準がコードに反映されていない
1. 該当要件カード（`docs/specs/[SPEC-ID].md`）を読み、何が変わったか確認
2. 実装ファイルを読み、変更が反映されているか確認
3. 反映されていない場合:
   - 軽微な変更 → 即座に実装を修正
   - 大きな変更 → build スキルで正式に対応

### 要件カードに存在するがコードにない機能
1. まず本当にコードにないか2段階探索で確認（構造ベース → grep）
2. コードにない場合:
   - **未実装:** build で実装する
   - **削除済み:** 要件カードから該当記述を削除する
   - **誤記:** 要件カードを修正する

## 4. サブエージェント関連

### サブエージェントが矛盾した結果を返した
1. 矛盾点を具体的に特定する（どのファイル/SPEC-IDについて食い違いがあるか）
2. 該当コードを自分で読み直して正しい方を判定する
3. 矛盾が解消できない場合:
   - 両方の結果をログに残す
   - 人間に判断を仰ぐ

### サブエージェントがコンテキスト超過で中断した
1. タスクをさらに小さく分割する
2. コンテキストファイルから不要な情報を削る
3. 「ファイル一覧の取得」と「内容の分析」を別のサブエージェントに分ける

## 5. 緊急修正後の要件カード同期忘れ

### 検知方法
- 緊急修正（コード先行）のコミットがあるが、対応する要件カード更新コミットがない

### 対処
1. 緊急修正コミットの差分を読む
2. 影響する要件カード（`docs/specs/[SPEC-ID].md`）を更新する
3. 「要件カード同期完了」のコミットを作成する

## 6. マージコンフリクト

### 要件カードのコンフリクト
- 異なる SPEC-ID の追加 → 両方残す（別ファイルのため自動マージ可能なはず）
- 同一 SPEC-ID の変更 → 両方の変更内容を読み、新しい受入基準を採用する

### コード・要件カードのコンフリクト
1. 両方の変更内容を読む
2. 片方が他方を包含している場合 → 包含している方を採用
3. 独立した変更の場合 → 両方をマージ
4. 矛盾する変更の場合 → 人間に判断を仰ぐ

### overview.md への複数人同時更新
1. 各自の変更がどのセクションに影響するか確認する
2. セクション単位で分離できる場合 → 両方のセクション変更を手動マージ
3. 同一セクション内の矛盾 → 後からコミットした方が全体を統合する責任を持つ

## 7. スキル固有の障害

### build / map の並列委任が中断した
1. 起動した各 coder / Explore の担当範囲（対象タスク・SPEC-ID）を確認する
2. 完了済みタスクの成果（マージ済みブランチ・返却サマリ）を確認する
3. 未完了タスクだけを対象に、[parallel-work](../../rules/parallel-work.md) に従って再委任する
4. 全タスク完了後、本体が横断レビューで整合性を検証する

### レビューループが3回超で収束しない
1. レビュー指摘内容を確認し、パターンを分類する:
   - **同じ指摘の繰り返し:** coder が指摘を正しく理解していない → 指摘を具体的なコード修正として書き直す
   - **矛盾する指摘:** レビュー基準が不安定 → 要件カードの受入基準・設計意図を明確化してから再レビュー
   - **些末な指摘の連鎖:** 重要度の低い指摘が続いている → 「Critical/Majorのみ対応」とスコープを限定
2. 3回で収束しない場合は現状のコードをコミットし、残りの指摘をIssueとして記録する

### スキル実行時に前提ファイルが見つからない
1. エラーメッセージから不足ファイルを特定する
2. 不足ファイルを生成する先行スキルを特定する（例: overview.md がない → map を実行）
3. 先行スキルを実行してから元のスキルを再実行する
