# スキル選択フローチャート

ユーザーのリクエストに対して、どのスキルを使うべきかを判定する。
skill-auditor がルーティング正確性を評価する際の参照表として使う。

## 新ハーネスの構成

全リクエストは **handle**（受け皿）が受け取り、内容に応じて即答／直接対応／5フェーズスキルへ振り分ける。
5フェーズ = grill（要件）/ mock（イメージ合わせ・見た目）/ plan（分解）/ build（自律実装）/ map（既存コード把握）。

| スキル | フェーズ | 守備範囲 |
|--------|---------|---------|
| **grill** | Grill | 要件ヒアリング。曖昧語・スコープを潰し要件カード `docs/specs/[SPEC-ID].md` に落とす |
| **mock** | Sketch | UIモックを作って見せる・既存画面の見た目だけ差し替え |
| **plan** | Plan | 要件カードから goal-board `impl-plans/[SPEC-ID].md` を作りタスク分解する |
| **build** | Build | 自律実装ループ（新規実装・仕様変更・緊急修正・テスト作成／補強） |
| **map** | Map | 既存コードを理解し軽量 overview 1枚にまとめる |

## 判定フロー

```
リクエストを受け取る（受け皿 = handle）
│
├─ 即答できる質問・雑談 → 直接回答（スキル不要）
├─ 調査・確認のみ（「どうなってる？」「確認して」）→ 直接調査して報告（スキル不要）
├─ 相談・壁打ち → 直接対応（スキル不要）
│
├─ 要件を固めたい（「○○を作りたい」「こういう機能がほしい」）
│  → grill（要件カードを作る）
│
├─ 見た目・イメージを合わせたい
│  ├─ 「画面モックを作って」「イメージを見せて」→ mock
│  └─ 「見た目だけ変えて」「このデザインに差し替えて」→ mock
│
├─ 進め方を設計したい（「タスク分解して」「マイルストーンに分けて」）
│  → plan（goal-board を作る）
│
├─ コードを書く・変える（実装を伴う）
│  ├─ 新機能の実装（「○○を実装して」「これを作って」）→ build
│  ├─ 既存機能の仕様変更（「○○の仕様を変えて」「バリデーションを変えて」）→ build
│  ├─ 緊急修正（「本番が落ちた」「至急直して」）→ build
│  └─ テストの追加・補強（「テストを書いて」「カバレッジを上げて」）→ build
│
├─ 既存コードを把握したい（「このプロジェクトを把握して」「構造を理解して」）
│  → map
│
└─ テンプレート管理
   ├─ 「テンプレートを更新して」→ sync-template
   └─ 「この改善をテンプレートに反映して」→ feedback-template
```

## よくある迷いポイント

### grill vs map
| 状況 | 使うスキル | 理由 |
|------|-----------|------|
| これから作る機能の要件を固める | grill | WHATを定義する（要件カード） |
| 既に動いているコードの構造を知りたい | map | コードから overview を起こす |
| 既存コードに新機能を追加する | grill → map | 追加分は grill で要件化、前提理解に map |

### mock vs build
| 状況 | 使うスキル | 理由 |
|------|-----------|------|
| 画面の見た目・レイアウトだけ変える | mock | ロジック不変、イメージ合わせに専念 |
| モックで方向性を確認したい | mock | Artifact でモックを作って見せる |
| 見た目の変更 + バリデーション追加 | build | ロジック変更を伴うため実装に入る |

### plan vs build
| 状況 | 使うスキル | 理由 |
|------|-----------|------|
| どう作るか計画を立てたい | plan | goal-board にタスク分解する |
| 計画に沿って実装する | build | goal-board を上から順に自律実装する |
| 小規模で分解不要 | build | 直接実装ループに入る |

### build が扱う範囲（旧スキルの統合先）
| 状況 | 使うスキル | 理由 |
|------|-----------|------|
| Specはあるがコードがまだない（新規実装） | build | 自律実装ループ |
| Specもコードも存在し、仕様を変える | build | 実装／仕様変更を1本で扱う |
| バグ修正・緊急修正 | build | 緊急修正も build 内で処理（コード先行を許可） |
| テストの追加・補強・作り直し | build | テスト作成も build の一部 |

### map vs grill（トリガーの見分け）
| 状況 | 使うスキル | 理由 |
|------|-----------|------|
| 「○○を作りたい」（未来の話） | grill | 新規要件のヒアリング |
| 「○○がどう動いてるか教えて」（現状の話） | map | 既存コードの理解 |
