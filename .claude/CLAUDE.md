## 開発ライフサイクル（5フェーズ）

| フェーズ | スキル | 役割 | 成果物 |
|---|---|---|---|
| Grill | `grill` | 要件を問い詰めて引き出す | 要件カード `docs/specs/[SPEC-ID].md` |
| Sketch | `mock` | UIモックで認識合わせ / 見た目差し替え | Artifact モック |
| Plan | `plan` | ゴール+マイルストーン分解 | goal-board `impl-plans/[SPEC-ID].md` |
| Build | `build` | 自律実装ループ（実装/変更/緊急/テスト） | コード + テスト |
| Map | `map` | 既存コード理解（オンデマンド） | overview `docs/overview.md` |

- 受け皿は `handle`（内容が不確定なリクエストを振り分ける）。
- **確定文書は要件カード1枚だけ**。詳細設計書は作らず、詳細はコードとテストを真実とする（旧BDUF型・大量Spec生成は廃止）。
- 並列実装は build が `.claude/rules/parallel-work.md` に従って worktree 分離で fan-out する（旧 orchestrate 3層PM は廃止）。

## Claude Code 設定

### ルール優先順位（複数ルールが同時発火した場合）

1. **security.md** — セキュリティは常に最優先。他と矛盾する場合はセキュリティが勝つ
2. **principles.md** — 行動原則・スキルルーティング・委任・スコープ確認・自律の封筒の既定（常時ロード）
3. **autonomous-mode.md / implementation.md / testing.md** — 実行時の規律
4. **その他のルール** — 状況に応じて適用

### セキュリティ設定

詳細は `.claude/rules/security.md` を参照（機密ファイル操作時に自動ロード）。
概要: サンドボックス有効、deny ルールで破壊的コマンドをブロック、PreToolUse フックで検証。

### モデル設定

Claude Code の `/model` コマンドで以下を推奨設定にすること：

- **プランモード（設計・判断）**: Opus
- **実行モード（コード生成）**: Sonnet

### テンプレート設計原則

`docs/templates/` 改修時は厳守：

- **1 テンプレート = 1 関心事**（要件カード=WHAT / goal-board=分解 / domain-knowledge=用語。混ぜない）
- テンプレートは軽量に保つ。詳細設計テンプレートは新設しない（詳細はコードとテスト）

## AI 主動開発ルール

行動原則・エビデンス検証・最小変更原則・完遂原則・スキルルーティング・Agent委任・スコープ復唱・出力方針・環境プリフライト・実装後レビュー・自律の封筒の既定は、**常時ロードされる `.claude/rules/principles.md` に集約**している。まずそちらを参照すること。以下はプロセス補足のみ。

### 新規プロジェクト（0→1）の開発順序

要件カード（全機能分）→ 認識合わせ（mock）→ 分解（plan）→ 実装（build）。途中で「実装して」と言われても、要件カードの確定を先に提案すること。

### goal-board（作業計画）

コード変更を伴う作業では、着手前に goal-board（`impl-plans/[SPEC-ID].md`、`docs/templates/goal-board.md` 形式）を作成する。PlanMode で作成してよい。

- マイルストーン（依存順・完了判定）、流用すべき既存コード、やらないこと（WHY NOT）、テスト計画を含める
- **セッション引き継ぎ**: コンテキスト圧迫時に新セッションで再開するための作業状態
- **Agent 委任**: coder に渡す「上から順にやるだけ」の指示書
- **レビュー時の照合**: 実装が計画通りか検証する基準
- 作業完了後は削除してよい（永続文書ではない）

### コンテキスト管理

- 自動圧縮が発生すると過去に読んだコードの詳細が失われ判断品質が低下する
- 判断精度の低下を感じたら、省略して畳まず goal-board に状態を書き出し、新セッション再開を提案する（コンテキストを守るために Agent に逃がすのではなく、セッションを分けて判断品質を維持する）

### マルチエージェント構成

実装を伴うリクエストでは `build` が必要時に自ら並列 fan-out する（`.claude/rules/parallel-work.md` に従い worktree 分離・取りまとめ検証）。旧 orchestrate（SuperPM→ドメインPM→PG の3層PM）は廃止。委任ワーカーは `coder`。
