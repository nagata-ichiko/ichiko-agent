# タスク分解パターン

build が goal-board のマイルストーンを実装可能な作業単位に分解し、独立なものを
[parallel-work](../../rules/parallel-work.md) に従って worktree 分離で並列 fan-out する際の共通手順。

## 1. 分析（コードには触れない）
- goal-board（`impl-plans/[SPEC-ID].md`）のマイルストーンと要件カード（`docs/specs/[SPEC-ID].md`）を確認し、影響範囲を特定する
- マイルストーンを coder に委任できる作業単位に分解する（1 coder = 1 作業単位）
- **共通基盤タスクの抽出**: 複数タスクが共通して必要とする基盤（共通UIコンポーネント、共通ユーティリティ、認証ガード、レイアウト等）を特定し、独立したタスクとして切り出す
- 依存関係を整理し、並列実行できるものと直列が必要なものを分ける
  - **共通基盤タスクは最優先で実行する**（他の機能タスクはこれに依存する）

## 2. 並列/直列の判定
- 相互に依存がなく、共通ファイル（lockfile・DBスキーマ・ビルド設定・共通コンポーネント等）で衝突しないタスクは**並列**（worktree 分離）
- 依存があるタスクは先行タスクの**完了を確認してから**着手する（**直列**）
- DBスキーマ変更を含むタスクは**並列禁止**（1本ずつ直列）
- 詳細な判定基準・フローは [parallel-work](../../rules/parallel-work.md) に従う

## 3. 委任コンテキストの渡し方
各 coder に渡す情報は prompt に含める（Agent はコンテキストをファイルパス経由でしか受け取れないため、参照すべきファイルはパスで渡し、coder 自身が Read する）。

### コンテキストの範囲
- **含めるもの**: 対象タスクに直接関係する要件カードの受入基準、依存する既存コードのパス、流用すべき実装のパス
- **含めないもの**: 他タスクの内部仕様、そのタスクが知る必要のない全体仕様

### 委任 prompt の必須要素
- **対象 SPEC-ID / タスク名 / 変更種別**（新規実装 / 仕様変更 / バグ修正）と 1-2 文の概要
- **依存関係**: このタスクの前に完了が必要なタスク（blockedBy）
- **参照すべきファイルパス**: 要件カード `docs/specs/[SPEC-ID].md`、流用元の既存コード、関連テスト
- **ブランチ名**: `feat/[SPEC-ID]-[概要]`（[parallel-work](../../rules/parallel-work.md) の命名に従う）
- **完了条件**: 該当マイルストーンの完了判定・受入基準（テスト GREEN を含む）

## 4. 完了確認
- 各 coder の完了は [共通仕上げ手順](finish-impl.md) で判定する（テスト GREEN・受入基準充足・設計⇔コード一致）
- 全 coder 完了後、本体が横断レビューを行い整合性を検証する（[parallel-work](../../rules/parallel-work.md) の取りまとめフェーズ）

### ブランチ同期ルール（並列実行時）

複数タスクが並列で実行される場合、ブランチの分岐元を制御して import エラーを防止する：

- **独立タスク（相互に依存がない）:** 同じ main ブランチから各自 `feat/[SPEC-ID]-[概要]` を分岐してよい（並列実行可能）
- **依存タスク（先行タスクが指定されている）:** 先行タスクが**mainにマージされた後**に `feat/[SPEC-ID]-[概要]` を分岐する。先行タスクのブランチから直接分岐しない（マージ前のコードに依存するとrebase地獄になる）
- **共通基盤タスク → 機能タスク:** 共通基盤タスクが全て main にマージされてから、機能タスクのブランチを作成する

```
main ─── 共通基盤マージ ─── AUTH マージ ──────────
              ↓                   ↓
         feat/共通基盤       feat/AUTH
                                  ↓ （AUTH マージ後に分岐）
                             feat/PROFILE
```
