# 実装ルール

ソースコード操作時に自動ロードされる。変更規模に応じてティアを判断し、該当ティアのルールを適用する。

| ティア | 判断基準 | 適用範囲 |
|--------|---------|---------|
| **Tier 1（小修正）** | バグ修正・文言・CSS・1-2ファイル | 必須事項 + コード探索(Step 2) |
| **Tier 2（機能実装）** | 新機能・既存変更・3ファイル以上 | 全セクション |
| **Tier 3（横断変更）** | 共通モジュール・アーキ変更・複数機能に影響 | 全セクション + 依存グラフ確認 |

`build` 経由は自動的に Tier 2 以上。直接のコード修正依頼は Tier 1 から判断。

## 必須事項（全ティア共通）

- 同画面・同種処理の既存パターン（命名・配置・変換）を把握し、一貫性を保つ
- 実装後は必ずテストを実行して全てパスさせる。通らなければ3回まで修正し、ダメなら失敗内容を添えて報告
- 修正後は変更前の値・名称でも grep し、旧記述が他ファイルに残っていないか確認する

## コード品質原則

- **Simple > Easy** — 理解しやすさ優先。暗黙の挙動より明示的な制御
- **DRY** — 3回以上の重複は共通化。ただし新規抽象化は事前提案（principles §3）
- **Fail Fast** — エラーは早期検出。曖昧なフォールバックを作らない
- **Why コメントのみ** — What/How は書かず「なぜこの判断か」だけ
- **関数〜30行 / ファイル〜300行** — 超過は責務分割を検討
- **Boy Scout Rule** — 触ったファイルは少し良くして返す（ただし依頼範囲を超える改変はしない）

### 禁止パターン（フォールバック濫用）

| 禁止 | 正解 |
|------|------|
| `user?.id ?? 'unknown'` | 呼び出し元で存在を保証 |
| 全呼び出し元が省略するデフォルト引数 | 必須パラメータにする |
| null合体で到達不能なデフォルト値 | パス検証で早期エラー |
| 空の catch ブロック | 上位へ伝播、または明示ログ |
| 条件非該当の黙殺（silent skip） | 明示的にログまたはエラー |

**例外**: 外部入力（ユーザー入力・外部API）のバリデーションでのフォールバックは許可。理由をコメントに記載。

### 設計情報が不足したときの既定判断

**大前提（Fixゲート）**: 通常の Build は要件カードの完成チェックリスト全達・未決=0 を前提に着手する（principles §1 / build 準備フェーズ）。よって受入基準・画面遷移・状態遷移・入力バリデーションは **Build 時点で不足しないはず**。実装中に不足に気づいたら、推測で埋めず **grill に差し戻す**のが原則。

以下の既定は、緊急修正（build モードC）等でやむを得ず先行実装する例外時の指針。適用したら**要件カード §8 未決へ追記し、事後に Fix する**：

| 不足 | 既定（例外時のみ） |
|------|------|
| バリデーション | 受入基準から導出。導出不能なら grill 差し戻し（例外時は一般的制約 + `// TODO: 仮採用`・未決へ追記） |
| エラーハンドリング | 既存パターンに合わせる。なければ標準HTTPステータス。上位レイヤーに集約 |
| 状態遷移 | 明記されない遷移は**禁止**（許可リスト方式）。表に無ければ実装しない |
| 削除方式 | 明記なければ論理削除（`deleted_at`） |
| 同時操作 | 明記なければ楽観的ロック（`updated_at` 比較） |
| リトライ | 明記なければ最大3回・指数バックオフ |
| キャッシュ | 明示指示がなければキャッシュしない |

例外時に仮採用した判断は作業報告に明記し、要件カードの未決へ必ず追記する。
