# 自律モードの実行規律

`build` スキルの自律ループ、および「質問せず進めて」と指示された長時間作業に適用する。
実ログ分析（713セッション）で、巨大スコープ×長時間ジョブが1ターン最大13回のコンテキスト圧縮を招き、
判断品質が低下し commit 未到達で成果ゼロ終了する事例が確認された。その再発防止が本ルールの目的。

## 1. スコープ分割（着手前に必ず）

- 巨大スコープをそのまま自律で走らせない。**着手前にサブタスクへ分割し `goal-board.md` のマイルストーンにする**
- 各マイルストーンに「完了判定（テストが通る/画面が動く 等の具体条件）」を書く
- 1マイルストーン = 1コミット単位を目安にする

## 2. 逐次コミット（最後にまとめない）

- **マイルストーン完了ごとに commit する**。全部終わってからまとめてコミットしない
- コミット前に該当範囲のテストを実行し、通ってからコミット（`.claude/skills/_shared/finish-impl.md` 参照）
- コンテキストが圧縮されても、コミット済みの成果は失われない

## 3. 長時間ジョブの待ち方（ポーリング禁止）

- docker build / E2E / 長いテストなど時間のかかるジョブは **`run_in_background: true` で起動し、完了通知で再開**する
- **`sleep` ループでのポーリングは禁止**（enforce-execution-rules.sh の sleep ブロックと整合）。待ちたい条件があれば Monitor を使う
- ジョブが失敗したら無限リビルドループに入らず、失敗ログを要約して原因を特定 → 1回直して再実行。直らなければブロッカーとして報告

## 4. コンテキスト節約

- **テスト・ビルド出力はファイルへリダイレクトし、要約だけを context に載せる**（全ログを貼らない）
- 大きなファイルは必要箇所だけ Read する（全文再読しない）
- 判断精度の低下を感じたら、省略して畳まず `goal-board.md` に状態を書き出して新セッション再開を提案する

## 5. ガードレールに当たったら

- deny/フック/サンドボックスに阻まれても、**回避策を自力で積み重ねない**
- 対応は2択のみ: **①1回だけユーザーに確認する** or **②ブロッカーありと明示してそのサブタスクをスキップ**し、他を進める
- フック・settings・ブランチ保護の一時無効化は禁止（security.md / principles.md §11）。これは `validate-command.sh` が hard-block する（`--no-verify`・`core.hooksPath`・`.claude/hooks`/`settings` の Bash 改変・chmod）ので、回避を試みず素直に手を止める
- 「一線」操作（外部アクセス/送出/不可逆破壊）はフックが**警告のみ**出す。警告が出たら実行前にユーザーへ1回確認する

## 6. 委任先の信頼性（build が fan-out する場合）

- 委任先には対象を 3-4 ファイル単位に絞って渡す（「Prompt is too long」は再起動でなく事前分割で回避）
- **委任先は主張を実行証跡（grep結果・生成ファイル）付きで返す**。未確認は「未確認」と明示し、在/不在を推測で断定しない
- **親は自己報告を鵜呑みにせず独立に機械検証してから採用**する（件数系はスクリプトで再計算）
- 並列完了は完了通知/Monitor で待つ。本体がポーリングや自前やり直しをしない（→ parallel-work.md）
