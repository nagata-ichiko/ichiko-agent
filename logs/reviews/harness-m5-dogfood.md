# M5 ドッグフーディング — spec-card v2 / Fixゲート / 遷移表 を実サンプルで検証

**日付**: 2026-07-01 / **対象**: M3.5 の固定フォーマット（spec-card v2・Fixゲート・遷移表）を実際に使用
**方法**: 全必須表を発火させるサンプル機能「記事コメントの投稿と承認」（UI＋状態＋入力＋権限を含む）を、grill の手順どおりに要件カード v2 へ埋め切る dry-run。埋める過程で生じた摩擦を記録する。

---

## 結論（3行）

- **到達目標は達成できる**: v2 の必須表を埋めると「読めばゴールが分かる」カードになり、遷移のブレは表で消える。フォーマットは実用に耐える。
- **摩擦を4件観測**: (1) §4画面遷移と§5状態遷移の責務重複、(2) 許可リスト方式で共通状態(loading/error)の行が増殖、(3) Fixゲート「未決=0」と“外部依存で今は決められない”論点の衝突、(4) §6入力表と§3受入基準のエラー系重複。
- **うち2件は即修正が有効**（テンプレに注記追加）。残り2件は運用注記で回避可。

## 検証: 埋めたサンプル要件カード（証拠）

```markdown
# [SPEC-COMMENT-001] 記事コメントの投稿と承認

## 0. 完成チェックリスト（Fix ゲート）
- [x] 目的が1-2行 / [x] 受入基準GWT / [x] 画面遷移表 / [x] 状態遷移表
- [x] 入力バリデーション表 / [x] スコープ外 / [x] 未決=なし

## 1. 目的
読者が記事にコメントでき、管理者の承認を経て公開される。

## 3. 受入基準（Given-When-Then）
- [ ] Given ログイン読者 When 本文を入力し投稿 Then コメントは「承認待ち」で保存され一覧には出ない
- [ ] Given 管理者 When 承認待ちを承認 Then コメントは「公開」になり記事下に表示される
- [ ] Given 管理者 When 却下（理由必須）Then コメントは「却下」になり投稿者にのみ理由が見える

## 4. 画面遷移表
| 遷移元画面 | トリガー | 遷移先画面 | 条件・ガード |
|-----------|---------|-----------|-------------|
| 記事詳細 | 「コメントする」 | コメント投稿フォーム | ログイン済み |
| コメント投稿フォーム | 投稿成功 | 記事詳細(トースト:承認待ち) | バリデーションOK |
| コメント投稿フォーム | 投稿失敗 | 同フォーム(エラー表示) | バリデーションNG |
| 管理コメント一覧 | 「承認」/「却下」 | 同一覧(状態更新) | 管理者 |

## 5. 状態遷移表
| 現在状態 | イベント | 次状態 | 副作用 | 許可アクター |
|---------|---------|-------|-------|------------|
| （新規） | 投稿 | 承認待ち | - | ログイン読者 |
| 承認待ち | 承認 | 公開 | 記事下に表示 | 管理者 |
| 承認待ち | 却下 | 却下 | 理由記録・投稿者へ通知 | 管理者 |
| 公開 | 非表示化 | 非表示 | 記事下から除去 | 管理者 |

## 6. 入力・バリデーション
| 項目 | 型/形式 | 必須 | 制約 | 違反時 |
|------|--------|------|------|-------|
| 本文 | text | 必須 | 1〜1000字 | フォームにエラー表示・保存しない |
| 却下理由 | text | 却下時必須 | 1〜200字 | 却下ボタン非活性 |

## 7. スコープ外
- 返信（ネスト）・いいね・通報。編集機能。

## 8. 未決
- なし
```

→ このカードは、実装後に「§4/§5/§6 と照合して正誤判定」ができる。目標（情報不足ゼロ・読めばゴールが分かる）は満たせた。

## 観測した摩擦と対応

### 摩擦1【即修正推奨】§4画面遷移 と §5状態遷移 の責務が重複する
「承認ボタン押下」は画面遷移（一覧→一覧）でもあり状態遷移（承認待ち→公開）でもある。どちらの表に書くか迷い、二重記載やブレが起きる。
→ **対応**: spec-card に「§4=画面の遷移（何が見えるか）/ §5=データ状態の遷移（何が変わるか）。同一操作は両方に現れてよいが、状態変化の真実は§5」と注記。**（今回テンプレへ反映）**

### 摩擦2【運用注記で回避】許可リスト方式で loading/error 行が増殖
「表に無い遷移は禁止」を厳格に取ると、各画面の loading/空/error 遷移を全部行にする必要があり表が肥大化する。
→ **対応**: spec-card に「共通状態（loading/空/error）は各画面の既定として1箇所にまとめ、画面ごとに列挙しなくてよい」と注記。**（今回テンプレへ反映）**

### 摩擦3【対応済み】Fixゲート「未決=0」 vs 外部依存で今決められない論点
「承認者は誰か（運用ロール）」等、ユーザーでも即答できずビジネス確認待ちになる論点がある。未決=0 を厳格化すると、無関係な部分の Build まで止まる。
→ **対応済み（2026-07-01）**: 「未決」を2分類に明記＝(a)解消必須=Build前にゼロ、(b)外部確認待ち=該当マイルストーンのみブロックし依存しない部分は Build 可。grill（Step4/5）・build（Fixゲート）・spec-card §0/§8・principles §1 に反映。

### 摩擦4【許容】§6入力表の「違反時」 と §3受入基準のエラー系が一部重複
重複は冗長だが、§6=項目単位の網羅・§3=シナリオ単位の受入、と粒度が違うので実害は小。現状維持。

## 判定

- M3.5 の固定フォーマットは**実用に耐える**。理想像（仕様だけで判断・質問不要）に対し、サンプルで再現性を確認。
- 摩擦1・2はテンプレに注記を追加（本ドッグフーディングで即修正）。摩擦3は次イテレーションで grill 明記を検討。
- 提案3（build 委任判断の決定化）は、今回のサンプルが単一マイルストーン中心で並列判断が発火せず未観測。実プロジェクト適用時に再観測する。
